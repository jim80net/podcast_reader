# Electron MVP — Design

## Context

Parent design: `docs/superpowers/specs/2026-06-11-desktop-packaging-design.md` (v3, "Electron app" section, security model F4/F9, process model F3/F7/N1, Phasing item 3). Phase 1 landed the engine contract this app consumes: pre-bound socket → persisted port in `engine-state.json` → atomic 0600 discovery file `{port, pid, token_fingerprint, version}` → `PODCAST_READER_READY` sentinel on stdout (`engine/process.py`); bearer auth on every route including health (`engine/app.py`); data dir = `PODCAST_READER_DATA_DIR` or `~/PodcastReader` (`engine/settings.py`). Phase 2 (stacked below this branch) landed `PUT /v1/keys` and the provider registry. The repo is Python-only so far; `app/` establishes the frontend conventions.

## Goals / Non-Goals

**Goals:**
- A working desktop app on Windows + macOS driving the real engine: submit, watch progress, read transcripts, configure settings/keys.
- Engine supervision exactly on the Phase 1 contract — no parallel discovery mechanism.
- Unsigned dev builds working end-to-end before any signing infrastructure exists.
- Packaging/update layout fixed so Phase 4 fills the engine payload without re-architecture.

**Non-Goals:**
- Download manager, CUDA/model/diarization packs, tools seeding (Phase 4).
- Chrome extension, pairing UI, cookie capture (Phase 5) — Settings ships without pack/cookie management sections.
- Floating video player (Reader hosts it later; no provisions beyond "Reader owns the artifact surface").
- Signed/notarized release pipelines (explicit user-blocking prerequisites; wired when credentials exist).

## Decisions

1. **Build tooling: electron-vite, TypeScript strict, no renderer framework.** electron-vite over plain esbuild: it solves the three-target build (main = node CJS/ESM, preload = isolated, renderer = browser) with one config, gives renderer HMR, and is the layout electron-builder docs assume — hand-rolling the same with esbuild is undifferentiated plumbing. The renderer uses vanilla TypeScript (typed DOM helpers + a hash router), no React/Vue: four views whose state is "render what the engine says" (job-record hydration + SSE patches), and the fewer lockstep-upgrading deps (Electron major churn is constant) the better. Revisit if the view count grows in later phases. Conventions: `app/` is an independent npm package (`app/package.json`, not a Python concern), `tsconfig` strict, `src/main/`, `src/preload/`, `src/renderer/`, shared request/response types in `src/shared/types.ts` mirroring `podcast_reader/types.py` (TypedDict ↔ interface, kept in sync by the e2e tests that exercise real payloads).
2. **Engine supervision = the Phase 1 handshake, mirrored in `src/main/engine.ts`.** The app resolves the data dir exactly like `settings.data_dir()` (env override, else `~/PodcastReader`). Startup: read `<data_dir>/engine.json`; if present and the PID is alive, read the token from `<data_dir>/engine-state.json` (same-user file, mode 0600) and `GET /v1/health`; adopt when the reported `token_fingerprint` matches both files. Anything else — dead PID, connection refused, 401, fingerprint mismatch — is a stale engine: kill the PID (graceful shutdown first when it answers health, then force), remove nothing (the new engine rewrites the file), spawn fresh. Never two engines. Spawn resolution order (the pre-Phase 4 dev posture, documented in README): ① packaged engine `<resourcesPath>/engine/podcast-reader-engine serve` when it exists, ② `PODCAST_READER_ENGINE_CMD` env override, ③ dev fallback `uv run podcast-reader serve` from the repo root. Readiness: watch child stdout for the `PODCAST_READER_READY` sentinel (printed strictly after the discovery file is on disk, per `write_discovery`), then read the discovery file — no port polling, no stdout parsing beyond the sentinel line.
3. **Quit sequence needs a portable signal: add `POST /v1/shutdown`.** POSIX SIGTERM reaches uvicorn's handler, but Windows has no SIGTERM — `child.kill()` is `TerminateProcess`, which skips `serve_engine`'s `finally` (children survive only via the Job Object, which degrades to best-effort inside nested jobs, per `_join_windows_job`). An authed `POST /v1/shutdown` that sets `server.should_exit` gives one graceful path on both platforms and runs the existing `finally` (kill_children → store.shutdown → remove discovery). App quit (and `before-quit` ahead of `quitAndInstall`): POST shutdown → wait for child exit (10 s) → force-kill (Job Object / process group backstop reaps children) → quit. An interrupted job is the designed outcome (`interrupted` + retry affordance), not a failure mode. The endpoint returns 202 before exiting (response races shutdown otherwise).
4. **Token and keys live in the main process only; the renderer is credential-free.** All engine traffic goes main-side (a typed `EngineClient`) and reaches the renderer over contextIsolated IPC (`ipcRenderer.invoke` for requests, `webContents.send` for events). This isn't just hygiene: the engine sets no CORS headers, so renderer-origin `fetch` would fail anyway, and SSE needs `fetch` + ReadableStream with an `Authorization` header (per the parent security model — no EventSource, no token-in-query). The main process consumes `/v1/events`, reconnects with backoff, and re-hydrates from `GET /v1/jobs` after every (re)connect — job records are the source of truth, the stream is an optimization. Preload exposes a narrow typed API (`window.api.*`); `nodeIntegration` off, `contextIsolation` on, `sandbox` on for the renderer.
5. **Key vault: safeStorage blobs in app `userData`, push-at-engine-start (F9/K1).** `vault.json` stores `{provider: base64(safeStorage.encryptString(key))}` — ciphertext at rest app-side, nothing key-shaped on the engine's disk. On every engine-ready (spawn or adopt) and on Settings save, main decrypts and `PUT /v1/keys` per provider; engine restarts therefore repopulate within one supervision cycle. Clearing a key in Settings pushes the empty string (the engine's documented "restore env fallback" semantics). When `safeStorage.isEncryptionAvailable()` is false (headless Linux dev), keys are held in main-process memory for the session and the Settings view shows a warning instead of silently writing plaintext. The "test key" button calls a new engine `POST /v1/keys/test {provider, api_key?}` — a minimal one-completion round-trip through the Phase 2 transport (`chapters.py`/`providers.py`), returning `{ok, detail}` with K4 redaction discipline (no response bodies, no key material). Engine-side over app-side testing keeps one provider code path: no TS duplicate of the registry, base URLs, or error mapping.
6. **Awaiting-confirmation becomes a real engine flow.** `POST /v1/jobs` gains `requires_confirmation: bool = false`; true journals the record in `awaiting-confirmation` without enqueueing. `POST /v1/jobs/{id}/confirm` transitions to `queued` and enqueues (409 from any other state); `DELETE /v1/jobs/{id}` discards — allowed only in `awaiting-confirmation` (409 otherwise; terminal-job deletion is not this change). Journal recovery already loads non-queued states without enqueueing, so pending confirmations survive restarts for free. Engine-side state (vs. holding pending URLs in the app) because the parent design lists `awaiting-confirmation` as an engine job state, it survives app crashes, and Phase 5's extension flow reuses it unchanged.
7. **Protocol handler: registered everywhere, trusted nowhere.** `podcast-reader://transcribe?url=...` registered via electron-builder `protocols` (NSIS registry keys, macOS `CFBundleURLTypes`) plus `app.setAsDefaultProtocolClient` in dev. Single-instance lock; URLs arrive via `open-url` (macOS) and `second-instance` argv (Windows). Handling: parse, require scheme `podcast-reader`, host `transcribe`, and an `http(s)` `url` param; reject anything else silently-with-log. Valid URLs are submitted with `requires_confirmation: true` and the window focuses the New view, which renders the URL and a one-click Run / Dismiss. Nothing protocol-initiated ever auto-executes.
8. **Reader renders the artifact in a sandboxed iframe, not a `<webview>` tag.** Main fetches `GET /v1/transcripts/{id}.html` with auth and hands the renderer the HTML string; the New/Reader view injects it via `iframe.srcdoc` with `sandbox="allow-scripts"` (the artifact carries an inline scroll-sync script when chapters exist — `html.py:589`), **without** `allow-same-origin`: the artifact runs in an opaque origin with no IPC bridge, no engine token, no parent access. Rejected alternatives: `<webview>` (Electron docs discourage it; needs `webviewTag` opt-in) and session-level `webRequest` header injection so an iframe could navigate to the engine URL directly (sprays the bearer token onto any renderer-initiated request and grants the artifact a real origin). The artifact is self-contained single-file HTML, so srcdoc loses nothing. The future video player mounts beside the iframe in this view.
9. **Packaging: engine as `extraResources/engine/`, full-download auto-updates.** The frozen onedir must live outside asar regardless of update strategy — executables can't run from an archive — and the spike's layout (`engine/podcast-reader-engine` + sibling `whisper-worker`, shared `_internal/`) maps directly onto `<resourcesPath>/engine/`. The differential-update question (parent design left it: uncompressed extraResources blockmap-friendliness vs. full-download) is decided here as **full-download for this phase**: app and engine version in lockstep and the engine (the bulk of the 200–250 MB) changes in essentially every release while the product is young, so differential transfers degrade to ~full size anyway (electron-builder #6265 is exactly this failure); blockmap tuning would add release-pipeline complexity for savings that only materialize once shell and engine release cadences decouple — revisit then (the extraResources layout already keeps that door open). electron-updater against GitHub Releases; NSIS per-user (no elevation, smoother updates) for Windows; dmg for macOS first-install plus the zip target electron-updater requires. Update flow: download in background, prompt, then the decision-3 quit sequence strictly before `quitAndInstall` — never replace files under a running engine. Phase 3 fixes this payload contract and wires a `--engine-dir` input into the build script; producing the release-grade frozen engine (packs, seeding) is Phase 4 — dev builds without an engine dir still work via the decision-2 fallback chain.
10. **Signing/notarization are user-blocking prerequisites, sequenced last.** Everything must work as unsigned dev builds (`electron-builder --dir`, ad-hoc NSIS/dmg) first. Separate tasks acquire a Windows code-signing identity (Azure Trusted Signing or OV cert) and an Apple Developer ID + notarytool keychain profile; only then are `win.signtoolOptions`/`mac.notarize` wired and tag-pipeline installer builds (Windows + macOS runners) enabled. macOS reality documented: unsigned dev dmg requires right-click-open / `xattr -d com.apple.quarantine`; auto-update on macOS requires a signed app (Squirrel.Mac refuses unsigned), so update e2e on macOS waits for credentials while the NSIS update path is testable unsigned.
11. **Testing: mock engine that honors the real handshake; one real-engine smoke.** The mock is a small TS HTTP server implementing the `/v1` surface with scriptable fixtures (job scripts that emit SSE event sequences, canned library/transcripts, settings echo, keys/test outcomes). Fidelity mechanism: a Playwright fixture creates a temp dir, writes `engine-state.json` (known token) and `engine.json` (mock's port/pid/fingerprint), and launches the app via `_electron.launch` with `PODCAST_READER_DATA_DIR` pointing there — the app exercises its production adopt path; no test-only branches in main. E2e flows: adopt; spawn-after-stale-discovery (mock killed, file left behind); four views; awaiting-confirmation confirm/dismiss; SSE reconnect/hydration; key push-at-start ordering; quit sequence (mock asserts shutdown POST before exit). The real-engine smoke (integration-marked) launches with the dev fallback (`uv run podcast-reader serve`), asserts sentinel → discovery → adopted health → Library renders — proving the TS mirror matches `process.py` reality. CI: a `node` job (typecheck `tsc --noEmit`, vitest unit, Playwright e2e under xvfb, real-engine smoke after `uv sync`); installer-build jobs deferred per decision 10.

## Risks / Trade-offs

- [Electron/electron-builder/electron-updater version churn] → exact-pin in `app/package.json`; upgrades are deliberate PRs; no renderer framework shrinks the lockstep surface.
- [TS mirror of `data_dir`/discovery/`JobRecord` drifts from Python] → the real-engine smoke test and e2e against real payload shapes are the tripwire; shared shapes live in one `src/shared/types.ts` with comments pointing at `types.py`.
- [`POST /v1/shutdown` is a remote-stop primitive] → localhost + bearer token, same trust boundary as `PUT /v1/keys`; 202-then-exit keeps clients un-hung.
- [Unsigned macOS builds can't exercise auto-update] → accepted; NSIS path covers update-flow testing until notarization lands (decision 10).
- [Full-download updates cost 200–250 MB per release] → accepted for lockstep releases; layout already extraResources, so switching to blockmap-friendly differential later is config, not architecture.
- [safeStorage unavailable on some Linux dev setups] → session-memory fallback + visible Settings warning; Linux is not a ship target (parent design).
- [Protocol URLs are attacker-controllable input] → strict scheme/host/param validation, awaiting-confirmation gate, URL always displayed before run; no shell/exec interpolation anywhere in the path.
- [Renderer compromise via artifact HTML] → opaque-origin sandbox (no `allow-same-origin`), no preload in the iframe, token never in renderer (decisions 4, 8).

## Migration Plan

Pure addition: new `app/` tree, three additive engine routes, one additive request field. Existing engine clients and the CLI are untouched; a Phase 2 engine without the new routes simply 404s — the app requires its own bundled/dev engine version (health `version` checked at adopt; mismatch → stale-kill-respawn). Rollback = revert PR; no data migration (vault file is new, ignored by older code).

## Open Questions

- Does adopt-version policy need more nuance than "mismatch = stale"? (A newer already-running engine from a parallel dev checkout would be killed; acceptable for v1?)
- Should `DELETE /v1/jobs/{id}` eventually cover terminal jobs (library hygiene), or stay confirmation-only until a real need appears?
