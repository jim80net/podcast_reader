# Chrome Extension — Design

## Context

Parent design: `docs/superpowers/specs/2026-06-11-desktop-packaging-design.md` (v3) — the "Chrome extension (MV3)" section (F3/F4/F10 + N1: fixed per-install port, user-mediated code pairing, fetch-stream progress with job-state hydration, `chrome.alarms` completion polling, protocol confirmation, no SW-resident SSE), the "Authenticated content" section (F2: extension-assisted cookie capture is THE novice auth path), and Phasing item 5 — the final phase.

Engine reality this builds on: fixed port + token in `<data_dir>/engine-state.json` (0600, `engine/settings.py`); bearer auth enforced by a single middleware in `engine/app.py` (`_require_bearer_token`), token never in query; job model with `requires_confirmation` / `confirm` / `DELETE` (job-pipeline spec); job runner snapshots settings at dequeue and currently feeds `cookies=os.environ.get("YT_DLP_COOKIES")` into `PipelineRequest` (`engine/process.py:190`); auth-failure hint is authored in `ytdlp.py:107` as CLI-shaped env advice. App reality: `protocol.ts` validates `podcast-reader://transcribe?url=` and submits with `requires_confirmation: true`; the New view lists awaiting-confirmation jobs with Run/Dismiss; `app/src/shared/types.ts` is the comment-pinned TS mirror whose parity the real-engine smoke asserts (key-set equality, per electron-app Q4); `app/tests/mock-engine/server.ts` implements the `/v1` surface with the real handshake.

## Goals / Non-Goals

**Goals:**
- One-click "transcribe this tab" from Chrome, with progress visible in the popup and a notification on completion — engine driven, app optional once paired.
- The F2 cookie path working end-to-end: failed auth → grant → capture → jar on the engine → retry succeeds.
- Pairing that an ordinary user completes in under a minute, with honest security posture.
- An explicit answer to the Q4 type-sharing revisit trigger.

**Non-Goals:**
- Firefox/Safari (different extension APIs; parent design out-of-scope).
- Chrome Web Store publication (store account is user-blocking; the build produces an uploadable zip, nothing more).
- Service-worker-resident progress streaming (parent design explicitly excludes it).
- Hosted inference (issue #10), token rotation UX beyond "re-run pairing" (parent design: rotation is rare and user-initiated; the engine grows no rotation endpoint here).

## Decisions

1. **`extension/` is an independent npm workspace, vite MV3 build, vanilla strict TS, no framework.** Same stance and rationale as the app renderer (electron-app decision 1): the popup is one small surface whose state is "render what the engine says"; the fewer lockstep deps the better. vite (not electron-vite) because the targets are a service worker + a popup page — plain `rollupOptions.input`, no Electron trio. vitest for unit; Playwright comes from the repo's existing harness.
2. **Type sharing (the Q4 adjudication): both TS consumers import the single comment-pinned mirror `app/src/shared/types.ts`; no codegen, no shared package.** The revisit trigger fired, so the options were weighed: (a) a shared npm package restructures `app/` packaging and adds workspace publishing mechanics for exactly two same-repo consumers — process cost, no correctness gain; (b) openapi-typescript codegen adds a build-time dep plus generated-artifact check-in/drift questions, and the existing mirror is already parity-enforced by the real-engine smoke (exact key-set equality of `JobRecord`/`LibraryEntry`/`EngineSettings`, per Q4) — codegen would re-derive what a test already proves; (c) a third pinned mirror in `extension/` is the one indefensible option — two hand-maintained copies of the same shapes. So: the extension's tsconfig includes `../app/src/shared/types.ts` (vite bundles it; no runtime coupling), new shapes from this change (pairing, cookies) are appended to the same file with the same comment-pinning, and the mirror's parity guarantee transfers to the extension for free. Escalation path stays documented: a drift incident or a third consumer (beyond app + extension) promotes the mirror to openapi-typescript codegen — that remains config, not architecture, because consumers already import from one path.
3. **Pairing flow (parent design N1, traced exactly).** App Settings → IPC → main `POST /v1/pair` (bearer-authed; only token-holders can mint) → engine returns `{code, expires_at}` and holds `{code, expiry, attempts}` **in process memory only** (never journal/settings/logs — the key-store discipline). The app displays the port and the 6-character code (unambiguous Crockford-style alphabet, no 0/O/1/I), plus a copy-pasteable combined form `<port>-<code>`. The user enters them in the popup; the popup calls `POST /v1/pair/claim {code}` at `http://127.0.0.1:<port>` — **the engine's single unauthenticated route**, exempted in the auth middleware by exact path match — and receives `{token}`; the popup verifies with an authed `GET /v1/health` and stores `{port, token}` in `chrome.storage.local`. Hardening: code is single-use (claimed or replaced by a newer mint), TTL 300 s, invalidated after 5 failed attempts, constant-time comparison, and a uniform 403 for wrong/expired/absent code (no oracle distinguishing "no pairing in progress" from "wrong code"). Threat analysis: same-user local processes can already read `engine-state.json` (0600) — claim adds them nothing; cross-user local processes can't read the file but can reach localhost, and the attempt cap (5 of ~10^9 codes) within a 5-minute, user-visible window makes brute force unviable. Because the port is fixed per install, reconnection is simply retrying it; pairing reruns only on token rotation or 401.
4. **Token storage threat model, stated honestly.** `chrome.storage.local` is unencrypted on disk inside the Chrome profile — readable by any process running as the user. That is the *same* exposure class as `engine-state.json` itself (0600 = same-user readable), so the extension does not weaken the existing boundary; an attacker with user-level disk access already owns the token at its source. What the design does defend: no content scripts exist (web pages never share a process with the token), the token travels only in `Authorization` headers (never URLs — uvicorn access logs), `storage.local` not `storage.sync` (the token must not ride Chrome Sync to other machines), and the popup/SW never hand it to page contexts.
5. **Least-privilege manifest.** `permissions`: `storage`, `alarms`, `notifications`, `contextMenus`, `activeTab`. `host_permissions`: `http://127.0.0.1/*` only — Chrome match patterns are port-agnostic, so the fixed per-install port is covered without wildcarding hosts. No content scripts, no `tabs` permission: a toolbar click grants `activeTab` (yielding the tab's URL), and context-menu clicks deliver `info.pageUrl` with no host access at all. The action is enabled on all pages (any yt-dlp-supported URL is fair game — same posture as the CLI); YouTube/X get no special grant because none is needed. `optional_permissions`: `cookies`; `optional_host_permissions`: `https://*/*` + `http://*/*` — *declared* broadly because Chrome requires declaration, but *requested* per-domain at capture time (decision 8).
6. **Confirmation semantics (adjudicated).** Trust derives from the channel, not the requester. The `podcast-reader://` protocol is unauthenticated and fireable by any web page — it cannot prove user intent, so it is confirm-gated (app-shell spec, unchanged). An extension submission is a deliberate click on a user-installed surface, delivered over the token-authed channel — exactly as trustworthy as typing into the app's New view, which submits `requires_confirmation: false`. So extension toolbar/context-menu submissions use `requires_confirmation: false`; "check the parent design" resolves cleanly because the parent's confirm requirement (F10) is scoped to *protocol-initiated* jobs. The two compose: when the engine is unreachable, the extension falls back to opening `podcast-reader://transcribe?url=<page>` — which launches the app *and* lands the job in `awaiting-confirmation`, correctly, because that submission traveled the unauthenticated channel.
7. **Progress model (parent design F3, verbatim).** Job state is the source of truth, not the stream. Popup open: hydrate from `GET /v1/jobs` first, then attach `fetch()` + ReadableStream on `/v1/events` with the auth header; the stream dies with the popup — by design. Popup reopened: hydrate again, re-attach. The SW tracks extension-submitted job ids in `chrome.storage.local` (bounded, most-recent-first); while any tracked job is non-terminal, a `chrome.alarms` alarm (30 s — Chrome's floor) polls `GET /v1/jobs/{id}` and fires a `chrome.notifications` toast on terminal state, then clears the alarm. The SW holds no SSE connection ever; every SW wake is stateless (read storage → poll → write storage).
8. **Cookie capture (F2), end to end.** Trigger: a tracked job fails with the new distinct code `download_auth_required` (decision 9); the popup renders a "Share your <domain> login" affordance. Flow: `chrome.permissions.request({permissions: ['cookies'], origins: ['https://<domain>/*', 'https://*.<domain>/*']})` — requested **at click time, for that domain only**; declined = no capture, manifest-time grants untouched. On grant: `chrome.cookies.getAll({domain})`, serialize to Netscape format (the documented `--cookies` jar shape: header line + 7 tab-separated fields, `#HttpOnly_` prefix for httpOnly cookies), `PUT /v1/cookies {domain, jar}`, then offer one-click resubmission of the failed source. The extension retains nothing: cookie values exist transiently in popup memory, are never written to extension storage, never logged.
9. **Engine cookie jars: persisted files, honest posture (the deliberate divergence from keys).** API keys are never-persisted because the engine consumes them in memory. Cookies cannot follow that discipline — yt-dlp takes `--cookies <FILE>`, so a file must exist; pretending otherwise (e.g. temp-file-per-job from a memory store) still puts the bytes on disk, just with a less inspectable lifecycle. So, stated plainly: jars persist at `<data_dir>/cookies/<domain>.txt`, written atomically with mode 0600 via the existing `_secure_write_text` machinery (dir 0700; on Windows, user-profile ACLs carry the protection — same note as `engine-state.json`), and live **until the user deletes or replaces them** — re-capture friction is precisely what F2 removes, and stale cookies degrade gracefully (yt-dlp auth failure → the same capture affordance). The compensating discipline matches keys where it can: `PUT /v1/cookies` validates the jar (parses as Netscape lines; every cookie's domain field suffix-matches the declared domain — no smuggling jars for other sites; size-capped) and stores it; `GET /v1/cookies` returns metadata only (`[{domain, created_at}]` — never cookie values); `DELETE /v1/cookies/{domain}` removes the file; jar content appears in no response, no log, and no diagnostic bundle (parent design: "no keys, no cookies"). Job integration: at dequeue the runner resolves the source URL's host against stored jar domains by suffix match (`host == domain` or `host.endswith('.' + domain)`); a match feeds `PipelineRequest.cookies` with the jar path, else the `YT_DLP_COOKIES` env fallback as today. CLI untouched.
10. **Phase-aware hints (N2) via a distinct error code.** `ytdlp.py`'s auth-detection currently raises `download_failed` with env-var advice baked in at the raise site — the wrong layer to know which face is running. Change: auth-required failures raise code `download_auth_required` with a neutral message; the hint is authored by the face that knows its affordances — the CLI prints the `YT_DLP_COOKIES` advice (unchanged copy), the engine job store maps the code to "share your login via the browser extension, or import a cookies file in Settings". Per N2, only now (Phase 5) may hints reference the extension; the broken Chrome/Windows `--cookies-from-browser` path is never recommended.
11. **App-side additions are Settings sections, not new views.** "Connect browser extension": button → mint via IPC → display port + code + expiry countdown (re-mint replaces). "Cookies": list domains from `GET /v1/cookies`, per-domain delete. Both ride the existing typed preload bridge; no renderer-side credentials change.
12. **Testing.** Extension logic is plain TS — vitest units for: URL eligibility, Netscape serialization (incl. `#HttpOnly_`), pairing client (claim → verify → store), hydration/merge logic, alarm scheduling decisions, storage wrapper. E2e: Playwright `chromium.launchPersistentContext` with `--load-extension` (the documented MV3 testing path) against the **reused** mock engine — `app/tests/mock-engine/server.ts` gains `/v1/pair`, `/v1/pair/claim`, and `/v1/cookies` routes plus `__mock` controls, keeping one fake for both consumers. E2e scope (justified): pairing happy path + wrong-code rejection, submit-from-tab → popup progress, popup close/reopen hydration, cookie *push* path with the permission pre-granted via Playwright's `browserContext` — the optional-permission **prompt itself is not automatable** (Chrome renders it outside the page), so the grant-decline branch is unit-tested and the prompt is a documented manual check. Engine side: pytest TestClient coverage for pair/claim (single-use, TTL, attempt cap, no-auth-required, uniform 403, nothing persisted) and cookies (validation, 0600, metadata-only listing, delete, jar-over-env precedence at dequeue, hint mapping). The app's real-engine integration smoke extends with one pair→claim→authed-health round-trip so the pairing contract is proven against the real engine, not just the mock.
13. **CI: one new `extension` job** mirroring the `node` job shape — npm ci, `tsc --noEmit`, eslint, vitest, vite build, then Playwright e2e under xvfb with the mock engine. Installer/store-upload automation: none (out of scope until a store account exists).

## Risks / Trade-offs

- [MV3 SW termination mid-anything] → SW is stateless by construction (decision 7): every wake re-reads `chrome.storage.local`; alarms re-fire; the popup owns all streaming. No correctness depends on SW longevity.
- [`POST /v1/pair/claim` is an unauthenticated localhost route] → single-use + 300 s TTL + 5-attempt cap + constant-time compare + uniform 403; minting requires the bearer token; analysis in decision 3 shows no new exposure class versus the 0600 state file.
- [Cookie jars are real credentials at rest] → 0600/ACLs, domain-scoped validation, user-visible list + delete, never read back, excluded from diagnostics; honest indefinite-retention posture documented in Settings copy (decision 9).
- [Mirror import couples `extension/` to `app/` source tree] → accepted in-monorepo; the import is one file with zero runtime deps; the escalation path (codegen) is pre-decided (decision 2).
- [Mock-engine pairing/cookie routes drift from the real engine] → the real-engine smoke's pair/claim round-trip (decision 12) plus pytest on the real routes bracket the mock.
- [Chrome minimums] → `chrome.alarms` 30 s floor requires Chrome 120+; `minimum_chrome_version` set accordingly in the manifest.
- [Popup closed during a permission request or capture] → capture is a single popup-lifetime transaction; if interrupted, nothing was stored anywhere; the affordance simply reappears (job still failed).
- [User pairs against a stale port after data-dir reset] → claim fails connection-refused or 403; popup's error path always offers re-pairing; the app always displays the current port.

## Migration Plan

Pure addition: new `extension/` tree, additive engine routes + one auth-middleware exemption, two Settings sections, one error-code split (`download_failed` → `download_auth_required` for the auth subset; consumers key on `{code, message, hint}` generically, and no spec pinned the old code). Rollback = revert PR; jars/pairings are new files/state ignored by older engines. An old extension paired with a newer engine keeps working (additive API); an unpaired extension against a pre-Phase-5 engine gets 404 on `/v1/pair/claim` and shows "update the desktop app".

## Open Questions

- Should the pairing display offer a combined `<port>-<code>` paste string only, or port and code as separate fields too? (Cosmetic; popup parses both either way.)
- Should `GET /v1/cookies` also surface a jar's cookie *count* or newest-expiry summary (still metadata) to help users judge staleness, or is `created_at` enough?
- Jar size cap value (proposed 256 KB) — large enough for any real site, small enough to bound abuse?
- Should the extension badge show a percent during `running` (step-derived) or just a spinner-ish state badge? (Pure UI; no API impact.)
