# Download Manager & Packs — Design

## Context

Parent design: `docs/superpowers/specs/2026-06-11-desktop-packaging-design.md` (v3) — sections "Frozen-bundle / downloadable-pack split" (F1), "Pack versioning" (F13), the tools-in-user-data and yt-dlp-self-update parts of "Operational accessibility details" (F6), and Phasing item 4. Evidence base: `spike/SPIKE_REPORT.md` (engine-extraction task group 5) — every mechanism here was validated or measured there; this design cites it rather than re-arguing it.

What exists today:

- **Engine** (`engine/{app,jobs,process,settings,library}.py`): bearer-authed `/v1` surface, single-worker FIFO job store with journal + SSE fan-out, atomic-write discipline (`atomic_write_json`, staged library commits), pre-bound-socket discovery handshake, child reaping (POSIX process groups / Windows Job Object).
- **Tool resolution** (`tools.py`): `resolve_tool` (external tools: explicit/env tools dir → frozen bundle `tools/` or interpreter sibling → PATH) and `resolve_bundled_worker` (frozen sibling executables; returns `None` unfrozen) — the two-class precedence the tool-resolution spec encodes. The spec explicitly defers "newer of bundled seed vs user copy wins" to seeding time, Phase 4 — that is this change.
- **Transcription** (`transcribe.py`): shells out to the `whisper-ctranslate2` console script — impossible under PyInstaller (no console scripts exist in a frozen bundle).
- **App** (`app/`): supervision via the discovery handshake, three-way spawn chain whose first preference is `<resourcesPath>/engine/podcast-reader-engine` (`engine-cmd.ts`), `dist.mjs --engine-dir` mapping a frozen onedir into extraResources, a scriptable mock engine + Playwright suites, and an integration smoke asserting TS/Python type parity.
- **CI**: `frozen-smoke` builds and boots the *spike* engine (`spike/build.sh`) — a placeholder this phase replaces with the real thing.

Constraints carried from the parent design and spike:

- A frozen app cannot acquire Python packages post-install; only shared libraries, executables, data files, and model weights can be downloaded later (F1).
- ctranslate2 4.8.0 pairs strictly with CUDA 12.x + cuDNN 9 (complete DLL set — partial sets fail at model load, faster-whisper #1279).
- Whisper model weights load offline from a local directory passed as `--model` with `HF_HUB_OFFLINE=1` (spike-proven).
- The diarization worker is GO at ~350–450 MB compressed / ~1.3 GB on disk, with the residual risk that its *freeze* was never built — the spike instructs scheduling that build smoke early.
- The signed install dir is immutable: nothing self-updates in place; user-data is the writable home for tools, runtime DLLs, models, and worker packs.

## Goals / Non-Goals

**Goals:**

- A packaged app whose first run takes a novice from installer to first transcript: detect hardware, recommend packs, download with visible progress, resume after interruption, and re-run from Settings.
- Production transcription inside the frozen bundle (whisper worker) with strictly better progress granularity than today's console-output parsing.
- The CUDA, model, and diarization packs installable, verifiable, version-validated at startup, and uninstallable.
- Managed tools (yt-dlp/ffmpeg/ffprobe) seeded to user-data with self-updating yt-dlp, so extractor breakage heals without app releases.
- CI proof that the **real** frozen engine boots, handshakes, installs a pack, and transcribes.

**Non-Goals:**

- Signing/notarization (Phase 3 user-blocking tasks; this phase keeps working unsigned).
- Chrome extension and cookie capture (Phase 5).
- A custom pack CDN: GitHub Releases (our built artifacts), PyPI (NVIDIA wheels), and Hugging Face (model weights) are the hosts; revisit only if they prove inadequate.
- Linux as a ship target (the CUDA pack is Windows-only per the parent design; the POSIX spawn-time `LD_LIBRARY_PATH` pattern is kept working for dev).
- GPU-accelerated diarization (the worker pack is the CPU torch build by design — GPU torch would double the pack).

## Decisions

1. **Pack downloads go through the engine — but on a pack installer, not the job queue.** Downloads are engine work, not app work, because the engine is the sole owner of the user-data layout and its atomic-write discipline (install = staging + verify + rename, exactly the library's pattern), because validation at install and validation at startup must be the same code, because the wizard, Settings, and a future `podcast-reader packs` CLI then share one implementation, and because the engine — which spawns the workers — is where hardware detection and compat knowledge already have to live. The app stays a thin renderer of engine state, per the established architecture. However, installs do **not** ride the `JobStore` queue: that store is deliberately single-worker FIFO, so a ~1 GB CUDA download would serialize against transcription jobs (and vice versa); job semantics (interrupted → retry) also mismatch download semantics (partial → resume). Instead a `PackManager` runs installs on its own thread (FIFO across packs, one transfer at a time), publishes progress events into the **same SSE fan-out** as job events (`GET /v1/events` — the consumer the app already runs), and `GET /v1/packs` is the hydration source of truth, mirroring the job-record/SSE pattern exactly. Pack state needs no journal: it derives from disk at startup (installed manifest present → installed; staging partials present → resumable; else absent).

2. **Pack hosting: no first-party hosting for anything we can fetch from its canonical source.**
   - **CUDA pack**: the engine downloads the *pinned* `nvidia-cublas-cu12` / `nvidia-cudnn-cu12` wheels straight from PyPI (sha256s pinned in our registry from the PyPI JSON API), then extracts `nvidia/*/bin/*.dll` — the complete cuDNN 9 set plus `cublas64_12`/`cublasLt64_12` — into `<data_dir>/runtime/`. This is the same channel NVIDIA itself uses, sidesteps us redistributing NVIDIA binaries at all (the user's machine fetches from NVIDIA's own PyPI artifacts), and the spike's EULA analysis (permitted, private-directory + attribution conditions) covers the install layout regardless.
   - **Model packs**: pinned Hugging Face snapshot files (`Systran/faster-whisper-*` repos: `model.bin`, `config.json`, `tokenizer.json`, `vocabulary.*`, `preprocessor_config.json`) by exact revision and per-file sha256, into `<data_dir>/models/<name>/`. GitHub Releases cannot host these anyway (2 GiB per-asset limit < large-v3's ~3 GB), and HF is the canonical distribution channel. Registry ships `tiny` (CI + low-end), `small`, `medium`, `large-v3`.
   - **Diarization worker pack**: the one artifact we build, hosted on **GitHub Releases** of this repo under dedicated pack tags (e.g. `pack-diarization-v1`) — measured ~350–450 MB compressed, comfortably under the asset limit.
   All three hosts support HTTP Range, so one resume implementation serves all packs.

3. **Manifests: registry-in-engine is authoritative; installed manifests record what's on disk.** The engine ships a static, versioned pack registry (`engine/packs.py` data): per pack — id, kind, display name, platform gate, download spec (URLs + sha256 + sizes), `component_versions` (e.g. cuDNN 9.x, model revision), `compat` (what this engine build requires, e.g. the ctranslate2 4.8↔cuDNN 9 pin), and license notices. A successful install writes `pack-manifest.json` (`{pack_schema: 1, id, version, component_versions, files: [{path, sha256, size}], licenses}`) into the pack dir **last** — a pack dir without a valid manifest is by definition not installed, which makes install atomic-by-construction and crash-safe without directory-rename gymnastics. Startup validation (per F13): for each installed manifest, check `pack_schema` and the registry's compat range; incompatible packs are flagged (surfaced as `incompatible` in `GET /v1/packs`, treated as absent by the pipeline) and the wizard/Settings offer the re-download delta. The registry is data + pure functions, unit-testable without network.

4. **Whisper worker contract: argv in, file out, line-protocol progress on stderr (the spike's recommendation, adopted verbatim).**
   ```
   whisper-worker AUDIO --model <name-or-dir> --device cpu|cuda \
       --compute-type int8|float16 [--language xx] --output-dir DIR
   stderr: "progress duration=<sec>" once after model load,
           "progress segment_end=<sec>" per segment
   stdout: absolute path of the written JSON on success
   exit:   0 ok / non-zero with human-readable stderr tail
   ```
   Rationale (spike §5): JSON-over-stdout buys nothing for multi-hour audio the engine re-reads anyway; file output matches the existing `transcribe()`/cache contract so `pipeline.py` keeps its shape; faster-whisper's segment iterator gives per-segment progress that maps 1:1 onto a new `step_progress` event — strictly better than parsing whisper-ctranslate2's console output. The worker emits `"words": null` for byte-shape parity (spike §2), calls `multiprocessing.freeze_support()` first (Windows re-exec safety), and on Windows calls `os.add_dll_directory(<data_dir>/runtime)` before importing `faster_whisper` iff the dir exists; on POSIX the engine sets `LD_LIBRARY_PATH` at spawn (the portable pattern — in-process mutation can't affect an already-running loader). The worker lives in-repo at `src/podcast_reader/workers/whisper_worker.py` (imports `faster_whisper` lazily; main package never imports it), with a thin `packaging/whisper_worker_entry.py` frozen entry script.

5. **Pipeline switch: bundled worker when present, today's shell-out otherwise.** `transcribe.py` first consults `resolve_bundled_worker("whisper-worker")`; when it resolves (frozen app), the worker contract is used — model name mapped to the installed model pack dir, stderr streamed line-by-line into `step_progress` events. When it returns `None` (unfrozen: dev, CLI, headless serve), the existing `whisper-ctranslate2` path runs unchanged, including HF auto-download and `--hf_token` diarization. This keeps the CLI and dev postures byte-identical to today while making the frozen path self-contained. Two policy choices on the frozen path: a **missing model pack is a structured job failure** (`model_missing`, hint pointing at Settings → Packs) — silently auto-downloading 3 GB mid-job is worse than a clear hint, and the wizard makes this state rare; a **cuda device without a usable CUDA pack (or without an NVIDIA GPU) degrades to CPU with a warning event** — degrade-don't-fail, consistent with chapters fault isolation, because a slow transcript beats a dead job. Compute type derives from the effective device (`float16` cuda / `int8` cpu), not a new setting.

6. **Diarization: separate frozen worker, dumb by design; the engine owns the merge.** Per the spike's GO verdict and merge-glue analysis: the engine pre-converts input to 16 kHz mono WAV using its managed ffmpeg (neutralizing the torchcodec/FFmpeg-shared-libs problem entirely — the worker reads WAV with stdlib `wave`+numpy), spawns `<data_dir>/workers/diarization/diarization-worker AUDIO.wav --output turns.json [--num-speakers N]`, and then performs the max-overlap speaker assignment (≈50 lines of stdlib interval math, unit-testable without torch) to enrich whisper segments with `speaker` before render. The enriched JSON is rewritten atomically in place (idempotent: segments already carrying `speaker` make the step a cache hit). `html.py` gains speaker-label rendering — without it the pack would be a no-op, since `html.py` consumes only `start`/`end`/`text` today. A new engine setting `diarize: bool` (default false) gates the step; it warns-and-skips when the pack isn't installed. The pack contains the frozen worker onedir **plus** a pre-seeded local HF cache (segmentation + embedding models + pipeline config — weights alone are insufficient), loaded offline; users need no HF account. **Cut-line (unchanged from the parent design)**: the worker's freeze is the one unbuilt mechanism, so its build smoke is scheduled first within the diarization task group; if it proves non-viable in size/effort, the group detaches to post-v1 (CLI diarization via whisper-ctranslate2 remains) without blocking this change's release.

7. **Tools seeding: reconcile at engine startup; versions compared via a seed manifest, not by executing binaries.** The engine build places yt-dlp/ffmpeg/ffprobe seeds plus a generated `tools-manifest.json` (`{name: version}`) into the bundle's `tools/` dir (where frozen `resolve_tool` already looks). At every engine startup, before serving: for each seed, if the user-data copy is absent or the seed manifest's version is newer than the recorded user-data version, copy seed → `<data_dir>/tools/` (atomic: temp + rename, preserving exec bits) and update the user-data manifest — implementing the tool-resolution spec's seeding-time newer-wins contract, which keeps the lookup-time invariant "the user-data copy is the current one". `yt-dlp -U` updates bump the recorded user-data version from `yt-dlp --version` output after a successful self-update. The engine exports `PODCAST_READER_TOOLS_DIR=<data_dir>/tools` for its own process when unset, so every existing `resolve_tool` call site picks up the managed copies with zero call-site changes. Failure posture: seeding errors log and continue (a broken seed must not prevent serving; PATH fallback still works).

8. **yt-dlp self-update: scheduled and failure-triggered, never against the install dir.** On engine startup (after seeding, in a background thread) the engine runs `yt-dlp -U` against the user-data copy when the last successful check recorded in the user-data tools manifest is older than 24 h. Additionally, when a job's download step fails in a way that smells like extractor breakage (yt-dlp exit ≠ 0 on a URL source), the engine runs `yt-dlp -U` once and retries the download **once**, emitting a warning event describing the self-update attempt; a second failure surfaces the normal structured error. Both paths run only when the resolved yt-dlp lives under the user-data tools dir (release binaries support `-U`; a PATH-resolved pip install does not — detected and skipped, keeping dev environments untouched).

9. **Hardware detection: an `nvidia-smi` probe, cached, surfaced with the pack listing.** At first `GET /v1/packs` (cached thereafter), the engine probes `nvidia-smi --query-gpu=name --format=csv,noheader` (PATH plus the standard `%SystemRoot%\System32` location on Windows, where the driver installs it); exit 0 with output → NVIDIA present + GPU names. The response carries `hardware: {platform, nvidia_gpu, gpu_names}` and per-pack `recommended` computed engine-side: CUDA pack recommended iff Windows + NVIDIA; model recommendation follows the parent design (large-v3 with GPU, small/medium CPU-only, Apple Silicon CPU int8); diarization never recommended by default (opt-in). One round-trip gives the wizard everything; no separate hardware endpoint.

10. **App setup UI: a wizard view gated app-side, packs rendered from engine state.** First run (app-side flag in the app's config, set on completion or skip) with the engine ready and recommended packs missing → the wizard view: hardware summary, recommended packs pre-checked with sizes, install with live progress (pack events over the existing forwarded SSE), resume offered for partial downloads, skippable (everything re-runnable from Settings → "Run setup again"). Settings gains a Packs section: each registry pack with state (`not-installed`/`installing`+progress/`installed`+version/`incompatible`/`failed`+error), install/uninstall actions, and license attributions from the manifests. IPC additions follow the established pattern: `packs:list`, `packs:install`, `packs:uninstall` invoke channels; pack events ride the existing `engine:event` push channel (the SSE consumer forwards them untyped through the same pipe; `src/shared/types.ts` mirrors the new event/status shapes and the integration smoke's key-set parity assertion extends to them).

11. **Packaging: `packaging/` at the repo root productionizes the spike; CI proves the real engine.** `packaging/` holds `engine.spec` (two Analyses — `engine_entry.py` calling the real `serve_engine`, `whisper_worker_entry.py` — one COLLECT/MERGE, the spike's proven mechanism), the two custom hooks (ctranslate2, faster_whisper — hooks-contrib still ships neither), entry scripts, and `build_engine.py` (resolves/downloads tool seeds, writes the seed manifest, runs PyInstaller, lays out `dist/engine/` exactly as `engine-cmd.ts` expects: `podcast-reader-engine[.exe]`, `whisper-worker[.exe]`, `_internal/` incl. `tools/`). A new uv extra (`worker`: faster-whisper) covers the worker's import; PyInstaller stays a build-tool install inside the build script's instructions, not a project dependency. `app/scripts/dist.mjs --engine-dir` consumes the output unchanged. CI: `frozen-smoke` is rewritten to build the **real** engine on an `ubuntu-latest` + `windows-latest` matrix, boot it with a temp data dir, complete the **authed** handshake (read token from `engine-state.json`, sentinel → discovery file → `/v1/health`), install the `tiny` model pack via `POST /v1/packs/model-tiny/install` (HF download, cached between runs), submit a 5 s fixture WAV job, and assert `done` with non-empty HTML — upgrading the parent design's "frozen-artifact smoke" from boot-only to end-to-end and exercising the download manager itself. The spike directory remains as evidence but CI stops building it. `release.yml` gains engine-build steps feeding `--engine-dir` (still publish-gated on signing, unchanged). The diarization pack build (separate venv with CPU-torch index, separate spec, HF gated-model fetch at build time) lands as a `packaging/build_diarization_pack.py` script plus a release-pipeline job that needs an `HF_TOKEN` secret — a user-blocking prerequisite.

## Risks / Trade-offs

- [Engine restart kills an in-flight download] → Range resume from `.part` files; state derived from disk so a restart shows `resumable`, and the wizard/Settings offer resume. Servers ignoring Range (none of the three hosts do) restart that file from zero.
- [Pinned NVIDIA wheel or HF revision disappears/changes] → sha256 verification fails closed with a structured pack error; registry updates ride app releases. Pins are exact (URL + hash), so a silent content swap is detected, not absorbed.
- [Trimmed cuDNN set temptation] → explicitly resisted: registry extracts the complete DLL set; the spike documents the incomplete-set failure mode (faster-whisper #1279). Size is the cost of correctness.
- [Diarization worker freeze was never built (spike residual risk)] → its build smoke is the first task in the group; the cut-line detaches the group without blocking release if it fails.
- [CPU fallback hides a broken CUDA install from the user] → the fallback emits a visible warning event naming the reason (no GPU / pack missing / pack incompatible), and Settings shows pack state; the job still completes.
- [yt-dlp self-update changes behavior under the engine mid-job] → updates run between jobs (startup thread) or inside the failing job's retry only; the binary is swapped by yt-dlp's own atomic self-replace, and only the user-data copy is ever touched.
- [Concurrent pack install during a running transcription job] → installs write only to staging until the manifest commit, and a job resolves packs at step start (an install completing mid-job is invisible until the next job); uninstall is refused (409) while a job is running or the pack is installing.
- [`GET /v1/events` becomes a mixed stream (job + pack events)] → events are self-describing by `kind`; existing consumers ignore unknown kinds by construction (the renderer store switches on kind); hydration endpoints stay separate (`/v1/jobs`, `/v1/packs`).
- [Windows CI runner cost/flakiness for frozen-smoke] → matrix limited to build+boot+tiny-e2e with cached HF model and cached uv; tiny is 75 MB. If flake dominates, the windows leg can pin to boot-only while ubuntu keeps the e2e — decided in the PR, not here.
- [GitHub Releases as pack host couples pack availability to repo visibility] → acceptable: the repo already hosts app releases; the registry abstraction (URLs in one data structure) keeps a host swap to a registry edit.

## Migration Plan

Pure addition. The engine API grows three routes and one event family; engine 0.3.0 + `MIN_ENGINE_VERSION` 0.3.0 bump together so the app's existing adopt-or-kill logic handles mixed versions. The frozen transcribe path activates only where `resolve_bundled_worker` resolves — unfrozen installs (CLI, dev, headless serve) are byte-identical to today. Settings gain `diarize` (default false) via the established merge-over-defaults upgrade. First seeding run populates `<data_dir>/tools/` without disturbing existing env-var workflows (`PODCAST_READER_TOOLS_DIR` set by the engine only when unset). Rollback = revert PR; installed packs are inert data under user-data that older engines simply never consult.

## Open Questions

- Should the windows leg of `frozen-smoke` run the full tiny-model e2e or boot-only (CI cost vs. coverage)? Default: full e2e; downgrade on demonstrated flake.
- Uninstall of the CUDA pack while `whisper_device=cuda` is configured: currently allowed (next job degrades to CPU with a warning). Should Settings instead force the device back to cpu on uninstall?
- Does the failure-triggered `yt-dlp -U` + retry belong in the CLI too, or stay engine-only? Current scope: engine-only (CLI users can run `-U` themselves); revisit on field reports.
- Model pack deltas: when a registry bumps a model revision, the wizard re-downloads the whole pack (no per-file delta). Acceptable for v1?
